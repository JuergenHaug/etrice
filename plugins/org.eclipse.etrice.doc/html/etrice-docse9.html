2.7 Traffic Light

2.7.1 Scope

In this tutorial you will build an example with a more advanced state machine. To make it more appealing we’ve added a little GUI with two traffic lights, one for the cars and one for pedestrians. The GUI also contains a button that can be used to request green for pedestrians.

You will perform the following steps:

  1. create a new model from scratch
  2. create a pedestrian light actor
  3. perform a first test
  4. implement the behavior
  5. do some configuration

2.7.2 Create a new model from scratch

The easiest way to create a new eTrice Project is to use the eclipse project wizard. From the eclipse file menu select File->New->Project and create a new Empty eTrice Java Project and name it TrafficLight.

2.7.3 Implement the PedestrianLight Actor

Since our GUI is connected using TCP/IP sockets we need to add an instance of the ATcpClient. We will explain the simple protocol used to control the GUI later as we go.

Again we first have to add an import statement.

1import room.basic.service.tcp.* from "../../org.eclipse.etrice.modellib.java/model/TcpService.room"

Then we can add the TCP client as socketClient. As we see the TCP client has two ports on its interface. One is for controlling the connection, the other one is used for the data. Therefore we add internal end ports to our actor and connect them (make sure to add the ports as conjugated):

Now we can begin to implement the behavior of our actor. In the initial state we try to connect our GUI application via a socket connection. So we create a State OpenSocket and an initial transition to it. In the entry code we use the Messages button and select open for the tcpControl port.

The code fragment

1tcpControl.open(data);

is inserted for us.

(We’ve already added a semi colon at the end of the statement.)

By inspecting the PTcpControl protocol we find that the open message takes an argument of type DTcpControl which in turn consists of a string - the IP address - and an integer for the port number.

So we complete the call to tcpControl.open() by inserting the connection data:

1tcpControl.open(new DTcpControl("localhost", 4443));

2.7.5 Implement the PedestrianLight Behavior

The ATcpClient returns established() if the socket connection is created and ready to use. So we use this trigger to make a transition into the operational state.

The switching of the lights will be implemented in a sub graph of the Operational state.

Now we turn to the sub state machine of Operational. We need to cyclically loop through all phases of the traffic light.

State Car Lights Ped Lights timeout [s]
AllRed red red 1
CarGreen green red n/a
CarYellow yellow red 1
CarRed red red 1
PedGreen red green 3

AllRed will be the initial state. There we start a timeout after which we do the transition to CarGreen. We will stay in this state until the request button is pushed which means that we can trigger on recive() from the tcpPayload port. Since nothing else will be sent back from our GUI we don’t have to look at the contents of the payload. The transition ends in CarYellow where we again start a timeout which will take us to the next state and so on until we again reach CarGreen which completes the cycle.

The next thing we need is a SAP for our timeout. We will call it ’timeout’. Then we can create the states and transitions in the sub state machine of Operational.

The only thing that is left to be done is the actual switching of the lights in our GUI. We have to send the commands as strings terminated by new line characters. The GUI reacts on the following commands:

  • "pedLights=red\n"
  • "pedLights=green\n"
  • "carLights=red\n"
  • "carLights=yellow\n"
  • "carLights=green\n"

To simplify the code we have to add to the entry codes of the states we propose to add two methods:

1Operation sendString(text: string) { 
2 "tcpPayload.send(new DTcpPayload(1, text.length(), text.getBytes()));" 
3} 
4Operation setLights(car: Light, ped: Light) { 
5 "sendString(\"carLights=\"+getCmd(car)+\"\\n\");" 
6 "sendString(\"pedLights=\"+getCmd(ped)+\"\\n\");" 
7} 
8Operation getCmd(light: Light): string { 
9 "switch(light) { 
10 case Light.RED: return \"red\"; 
11 case Light.GREEN: return \"green\"; 
12 case Light.YELLOW: return \"yellow\"; 
13 default: return \"\"; 
14 }" 
15}

Since we use data types we also need to import the Types.room from the modellib.java:

1import room.basic.types.* from "../../org.eclipse.etrice.modellib.java/model/Types.room"

Finally we add entry codes to the five states. For AllRed e.g. we have

1State AllRed { 
2 entry { 
3   "setLights(Light.RED, Light.RED);" 
4   "timeout.startTimeout(1000);" 
5 } 
6}

And so on for all states corresponding to above table.

2.7.6 Configuration

We want to use this example to illustrate how instance data can be configured using a configuration model. We hard wired IP address and port into our model which of course is not very nice.

To use configurable data for our purpose we have to add a new attribute to out PedestrianLight actor structure:

1Attribute ipConfig: DTcpControl [ "configuration of the IP-port for the communication with the Traffic Light GUI" ]

Then we add a configuration model to our model folder. The contents of this model has to be

1ConfigModel TrafficLight { 
2 import TrafficLight.* from "TrafficLight.room" 
3 
4 ActorInstanceConfig TrafficLight/main/pedLight { 
5   Attr ipConfig { 
6    Attr IPAddr="localhost" 
7    Attr TcpPort=4443 
8   } 
9 } 
10}

This way we can configure specific attributes of actor classes or instances.

Now we only have to add this model to our launch configuration for the generator, re-generate and we are done.

As an exercise to the reader we leave the following task. Add a second PedestrianLight to the application and connect it to a second GUI (listening on another port of course).
